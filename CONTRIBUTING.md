# Contributing to Core-Monitor CLI

## Overview

Core-Monitor CLI is developed with a focus on reliability, clarity, and minimal overhead.

Contributions are welcome. To ensure consistency and maintainability, all contributions are reviewed and must align with the project’s design and implementation principles.

## Before You Contribute

Before submitting changes, please:

- review the existing codebase and structure
- check for existing issues or discussions related to your contribution
- ensure that your proposed change aligns with the goals of the project

If the change is substantial, consider opening an issue first to discuss the approach.

## Contribution Guidelines

### Code Quality

Contributions should:

- be clear, concise, and maintainable
- follow the existing coding style and structure
- avoid unnecessary complexity or abstraction
- prioritize correctness and performance

Changes that introduce unnecessary dependencies or significantly increase complexity may not be accepted.

### Scope

Contributions should remain focused.

Avoid combining unrelated changes in a single pull request.

Features that significantly expand the scope of the project may require prior discussion.

### Security and Safety

Changes involving:

- privileged operations
- system-level interaction
- configuration handling

must be implemented carefully and should not introduce unsafe or unintended behavior.

## Submitting Changes

To submit a contribution:

1. Fork the repository
2. Create a dedicated branch for your change
3. Implement your changes
4. Ensure the project builds and functions as expected
5. Submit a pull request with a clear description of the change

Pull requests should include:

- a concise summary of the change
- the reason for the change
- any relevant context or tradeoffs

## Review Process

All contributions are reviewed before merging.

During review:

- feedback may be provided
- revisions may be requested
- changes may be declined if they do not align with project goals

The review process is intended to maintain consistency and quality across the codebase.

## Acceptance Criteria

A contribution is more likely to be accepted if it:

- improves correctness, stability, or performance
- aligns with the project’s existing architecture
- maintains a minimal and focused design
- does not introduce unnecessary complexity

## Out of Scope

The following are generally not accepted:

- large feature additions without prior discussion
- stylistic rewrites without functional benefit
- experimental or unmaintained code paths
- changes that conflict with the project’s direction

## Code of Conduct

Contributors are expected to interact respectfully and constructively.

## Attribution

By contributing, you agree that your contributions may be used, modified, and distributed as part of the project under its existing license.
